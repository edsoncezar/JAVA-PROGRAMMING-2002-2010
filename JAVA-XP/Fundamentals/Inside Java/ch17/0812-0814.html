gies for Java

Other non-relational DBMS-based options may be available to Java application programmers who want to introduce persistence to their applications. The options presented in the following sections are Persistence through Object Serialization and ODMG Java Object Database Standard.

NOTE
Objects that exist beyond the life of the program that created them are called persistent objects.

Object Persistence Using Object Serialization

For small applications, the amount of data involved may not be sufficient to warrant the use of a DBMS. In such situations, it may be optimal to use Object Serialization to emulate database functions such as persistence.

Object Serialization involves mapping objects into an object output stream and then reconstructing these objects from an object input stream.

NOTE
Serialization has been mainly utilized for serializing objects into a byte stream for communication via sockets in Java Remote Method Invocation.

Most common Java objects can be serialized by using the Object Serialization system (http://www.javasoft.com/products/JDK/1.1/docs/guide/serialization/index.html). Serialization of user-defined classes requires the implementation of either the Serializable or Externizable

Page 813

interface. The difference between these two interfaces is that the object stream for Serializable objects contains sufficient information for reconstructing an object from the stream, whereas Externizable objects are solely responsible for their external formats and reconstruction.

Serialization can be used for persistence by creating files of objects. These files can be used instead of databases, but it becomes the responsibility of the programmer both to save all the objects in files (using serialization) before quitting the program and
to read these objects from files before starting execution again. The overheads involved are quite considerable and increase rapidly with growing numbers of application objects. An example of providing persistence for Java objects is Persistent Java (http://copeland.smartchoice.com/~laforge/index.html#Introduction), created by Bill la Forge. Serialization has been used for persistence in other languages, such as C++, using products such as the Tools h++ Class Library from RogueWave Software, Inc. (refer to http://www.roguewave.com/products/tools/toolsfund.html).

ODMG Java Object Database Standard

Object Database Management Group (ODMG) and JavaSoft created a new working group in February 1996. The goal of this working group is to define an object database standard for Java. Object databases are ideally suitable to represent Java objects in a transparent and seamless manner. Key concepts of the standard include:

  • Transparent persistence
  • Persistence by reachability
  • Multiple implementations of Java binding
  • Addition of classes to Java to implement the complete ODMG object model

Transparent persistence involves making active instances of Java objects persistent without modifying existing code. Persistance by reachability requires that all objects that can be reached from any root object be made persistent at commit-time. The Java environment must be enhanced to fully comply with the ODMG object model. This requires that classes and constructs such as collections, relationships, transactions, and databases be added to Java.

Currently, some products are designed for object-database support for Java. These products include ObjectStore PSE (persistent storage engine) from ObjectStore and POET for Java from POET. These products, however, do not claim to be compliant with the new ODMG Java standard for object databases.
