DELETE

The DELETE reserved word is used to construct statements that delete one or more rows from a given table. The correct syntax follows: 

DELETE <table name>
WHERE <condition>

Let's return to the example of Professor Simmons. Suppose that one day the Professor decides to leave academia to pursue a career producing workout videos. Given this situation, all courses taught by him would have to be removed from the COURSES table. The correct SQL to remove these rows follows: 

DELETE Courses
WHERE Professor = "Simmons"

TIP
When dealing with this statement, remember that it deletes an entire row. To delete individual cells in a given row, use the UPDATE keyword to change the value of the cell in question to a blank value.

AND

In the above sections on the various SQL reserved words, all operations were performed on a single table. This, however, ignores the fact that relations between tables are maintained. For this reason, the AND reserved word is used to perform operations on two or more tables. Remember that the beginning of this chapter introduced the example of selecting the names of all students who had received an A in the course with ID 00042. By using both the SELECT and AND reserved words, the correct SQL to extract all student names who received an A in the course with ID 00042 follows: 

SELECT Students.Name
FROM Students, Grades
WHERE Grades.Grade = "A"
AND Students.StudentID = Grades.StudentID

The preceding example introduces two new concepts: First, the management of multiple tables. Secondly, the use of multiple conditions in the WHERE clause. Perhaps most noteworthy about this statement is the new notation used when specifying columns. Because column names must only be unique for a table rather than the entire schema, it is necessary to indicate the exact table to which each column belongs. It is for that reason that the notation <table name>.<column name> is used.

One final note on the AND reserved word: Some situations exist where this word is used for operations on one table. To select, for example, all courses where the student with ID 00011 received an A, the following SQL is used: 

SELECT CourseID
FROM Grades
WHERE Grade = "A"
AND StudentID = 00011

To expand on this example, use the following SQL to select the names of the courses where the student with ID 00011 received an A: 

SELECT Courses.CourseName
FROM Courses, Grades
WHERE Grades.Grade = "A"
AND Grades.StudentID = 00011
AND Courses.CourseID = Grades.CourseID

Now that we've covered the basics of databases, it's time to relate databases to the topic at hand, JDBC.

Introducing JDBC

As the beginning of this chapter indicated, Java's cross-platform capabilities lead one to realize that some technology for accessing relational databases from a Java application (or applet) is necessary. This technology is the Java Database Connectivity Standard (JDBC).

What is JDBC?

JDBC is intended as a call-level SQL interface that provides Java programmers with a uniform interface to a wide range of relational databases. JDBC is similar in concept to Microsoft's Open Database Connectivity (ODBC) API, which has become the standard for relational database access. Because virtually all database engines offer access using

Page 775

the ODBC standard, and because the JDBC standard enables Java code to perform ODBC calls, it is easy to understand why JDBC is preferred over other non-open propitiatory standards.

What Does JDBC Do?

As you noted when studying SQL at the beginning of this chapter, SQL is an excellent language for managing data when stored in a series of tables. The primary function of JDBC is managing data stored in tables. To fulfill this function, JDBC defines Java classes to represent such elements as database connections, SQL statements, result sets, and metadata. These classes and interfaces are written in Java, so they provide a natural interaction with relational databases.

JDBC Release Information

The JDBC 1.1 version is currently available. This version, included in JDK 1.1 (Java Development Kit), can be downloaded as a compressed .tar file or a .zip file from the following Web page: 


http://splash.javasoft.com/jdbc/index.html

Although this site includes the .java and the .html files for the API, it does not include any database drivers. A database driver acts as a middleman between your executing Java code and the relational database. For JBDC to work properly, the relational database being used must either provide a native implementation of JDBC or there must be a driver available for that database in question. The topic of database drivers is elaborated on later in this chapter. The JDBC 1.1 specification is available from the JavaSoft site as a postscript or acrobat file.
