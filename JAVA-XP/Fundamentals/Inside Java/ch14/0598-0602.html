NOTE
The difference between low-level events and semantic events can be confusing. The basic difference is that low-level events have no semantic meaning associated with them, and semantic events have a semantic meaning associated with them. For example, when you click a check box with a mouse, several events are emitted. Two of these events are MouseEvent.MOUSE_CLICKED which is a low-level event, and ItemEvent.ITEM_STATE_CHANGED which is a semantic event. The MouseEvent.MOUSE_CLICKED event is a specific event that happens to a GUI component, but has no semantic meaning associated with it. The ItemEvent.ITEM_STATE_CHANGED event has a semantic meaning associated with it: "the state of an item has changed." It is not tied to a specific low-level user interface event, rather it can be triggered by a clicking the check box or pressing the space bar when the check box has keyboard focus. Semantic events are useful because you don't have to be concerned with all of the low-level events that can produce the higher-level, semantic event.

Low-level and semantic events are grouped and organized into a hierarchy of event classes, which this chapter will discuss after examining the event listener classes.

The Event Listener Interfaces

Nine event listener interfaces exist: Three listen for semantic events and six listen for low-level events. The java.awt.event package includes each of these listener interfaces, which are derived from the java.util.EventListener interface.

The three semantic event listeners are:

  • ActionListener
  • AdjustmentListener
  • ItemListener

The six low-level event listeners are:

  • ComponentListener
  • FocusListener

  • KeyListener
  • MouseListener
  • MouseMotionListener
  • WindowListener

Let's examine these event listeners one at a time:

ActionListener

An ActionListener listens for the ACTION_PERFORMED event type described previously. This event listener has one method, which is called when an event source registered with this listener fires an ACTION_PERFORMED event type. This method is the actionPerformed(ActionEvent).

Notice the similarity between the event type "ACTION_PERFORMED" (which is actually the name of the constant that the ActionEvent class uses to identify this event type) and the method name "actionPerformed." This similarity is repeated for all event types with corresponding event listener methods.

The argument passed to all the event listener methods is an instance of the event class that contains the event type that was fired. The listener method can call methods of this object to gain more information about the event. For example, the getSource() method returns the event source, which is the component that fired the event.

ComponentListener

A ComponentListener listens for event types contained in the ComponentEvent class. This event listener has four methods, each of which correspond to an event type in the ComponentEvent class. These methods are:

  • componentHidden(ComponentEvent)
  • componentMoved(ComponentEvent)
  • componentResized(ComponentEvent)
  • componentShown(ComponentEvent)

The event types to which they correspond are COMPONENT_HIDDEN, COMPONENT_MOVED, COMPONENT_RESIZED, and COMPONENT_SHOWN.

Although the AWT automatically handles moving, resizing, showing, and hiding components, you can override these methods to notify you as these events occur. For more information on ComponentListener, refer to the following Web page:
