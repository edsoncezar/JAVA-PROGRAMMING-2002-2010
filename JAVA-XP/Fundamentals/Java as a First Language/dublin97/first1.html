Getting Started

 

This part of the tutorial will describe the process of getting started. (This is not strictly true - an initial few weeks spent accommodating to the environment and learning how to use the compiler by the entry of some initial Java class files, including client/ class designs is recommended. However time is short today …).

The first class hierarchy which is met should be a model of a readily comprehensible real world object. This would preclude hierarchies such as the BankAccount hierarchy which are models of objects whose real-word existence is essentially abstract. A ClickCounter is being modelled here - the real-world analogue being the small hand held mechanical devices which night club bouncers use to count the number of people entering.

This approach is stressing the extendibility of a class from the outset and is imparting a fundamental message of development by extension. The process of developing the hierarchy also introduces the design, build, demonstrate (test) … technique from the outset.

One alternative approach might stress the use of existing classes, or hierarchies, at the outset; concentrating upon the development of client programs before considering the construction of classes.

Another alternative approach might stress monolithic classes at the outset, moving onto development by extension only after they have been assimilated.

Whatever approach is used there is clearly a need for other real-world based class hierarchies to be suggested, rather than the tired old clichés which are beginning to permeate introductory text books. One alternative which has been used at SBU is the (petrol)Pumps hierarchy.
