The RoomMonitor enquiry actions

The three enquiry actions are implemented as follows. 

0041     public int numberCurrentlyInRoomIs() { 
0042        return  theCounter.numberCountedIs();
0043     } // End numberCurrentlyInRoomIs.
0044  
0045  
0046     public int maxEverInRoomIs() { 
0047        return maxEverInRoom;
0048     } // End maxEverInRoomIs.
0049  
0050  
0051     public int totalNumberEnteredIs() { 
0052        return totalNumberEntered;
0053     } // End totalNumberEnteredIs.

The latter two of these return the value of the appropriate encapsulated int variable. The numberCurrentlyInRoomIs() action returns the value obtained from theCounter's numberCountedIs() action. This value will be maintained by the calls of count() and unCount() associated with the enterRoom() and exitRoom() actions.

The intention here is to emphasise that a class should perform its actions, where possible, by making use of existing classes either by extending them (as above) or encapsulating an instance (as in this example). This introduces the distinction between an is a and a has a relationship between classes and their instances. Where a class can safely inherit all existing actions then an is a relationship is suggested. Where the inheritance of an action is not safe then a has a relationship is suggested. In this example if it were to extend the WarningCounters class then the count() action would either have to be overridden, which is not an appropriate action for a RoomMonitor, otherwise a client could call count() directly and so invalidate the values of maxEverInRoom and totalNumberEntered.
