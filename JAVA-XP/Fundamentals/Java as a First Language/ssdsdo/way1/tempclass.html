The Temperature class diagram

The first class that will be developed is called Temperature and can be used to represent a temperature in centigrade, accurate to one tenth of a degree.

There is no indication of why this class is needed. It may be needed for:

recording a patient's temperature in the software that is controlling some medical equipment
or as a part of an oven a fridge or a central heating system
or is part of a weather forecasting system.
or ...

The intention of Object Oriented programming is to make the best possible model of something in the hope that it can be used in a number of different ways.

 

All of these temperatures would be modelled by different instances of the Temperature class. This is similar to saying that all people are humans.

As everyone is human we all share some things: how to think, how to breathe, etc. But as we are all individuals we are also different: different eye and skin colours, heights, names, etc. In object orientated terms everyone is an instance of the Human class.

The class diagram for the Temperature class is as follows.

A class diagram indicates what is being modeled by the class and what can be done with it.

This top part of the diagram indicates that the class called Temperature is an extension of a class called Object. All classes in Java are either extended directly from Object, or from a class which extends Object, or from a class which extends a class which extends Object etc.

As (almost) everything in Java is an instance of a class which somehow extends Object, (almost) everything in Java is an Object. This concept of is a, provided by extension, is one of the most fundamental ideas of Object Oriented Programming.

The main part of the class diagram is divided into three regions.

  • On the right: anything which crosses the right hand side of the diagram is fully public.
  • In the middle: anything which does not cross any side of the diagram is fully private.
  • On the left: anything which crosses the left hand side of the diagram is partially public.

Anything which is fully public can be seen and used by any other thing that can see the class. Anything which is fully private can be seen and used only by the class itself. There are various possible degrees of partially public visibility, the simplest is that it can be seen only by classes which extend.

The only private resource is called theTemp and is an attribute of the class. It is this thing that actually stores the number that is used to represent the temperature. It is private in order to protect it.

The first two public resources both have the same name as the class, this indicates that they are constructors and are used to make sure that the instances are created in a well-defined initial state.

The first constructor has no arrows pointing to it, which means that it requires no arguments, and is the default constructor. It will ensure that the Temperature instances that it constructs will represent the temperature 0.0oC.

The second constructor has a single arrow pointing to it, which means that it takes a single argument called setTo. It will ensure that the Temperature instances that it constructs will represent the temperature supplied to it as an argument.

The remaining public resources are both methods.

Both of these methods have a single arrow leaving them which indicates that they are inquiry methods and allow something about the private nature of the class to be known

The first method is called getTemp() and will allow the value of the private attribute theTemp to be known outside the class.

The second method is called toString() and will produce a representation of the temperature, for example "37.6 degrees centigrade". All classes should supply a toString() method to replace (override) the toString() method in the Object class.

The only partially public resource is a method called setTemp(). This method has an arrow pointing to it, which means that it takes a single argument called setTo and changes theTemp to that value. A method which changes the value of a classes attribute is known as a setter.

  

Class diagrams are one of the most fundamental notations that are used in the design of Object Oriented programming. It is essential that you are able to read them fluently. In order help with this you should write out a list of all the bold italic terms in this document and explain what they mean in your own words.

Having started to explain class diagrams and instances, an initial definition of an Object can be given.

An object is an instance of some class
and has state and behaviour.

The state of an instance is the combined value of all of its attributes, its behaviour is defined by the methods shown on its class diagram.
