Interface Design
Best Practices in Object-Oriented API Design in Java
by Bill Venners

This book is now called API Design.

Interface Design is a book-in-progress. To read online versions of the book, which I'll be posting as I write it, visit the Table of Contents.

Object Design is a companion book that contains patterns and idioms for designing Java objects.

Most Recent Update: Friday, 11/9/2001 - Renamed the Interface Design project to API Design. I have spent probably 2 solid months this year getting a build system going for book writing, including buying and learning to use FrameMaker, plus developing many tools in Java and Python to help me process code and generate nice web pages for the book projects. I'm leaving the older stuff in this area for a while, until the new one (which comes out of FrameMaker) catches up and passes the old one.

Wednesday, 9/13/2000 - Added pages for more guidelines, using primarily handout material from my Object Design Workshop. See the Table of Contents. Here are the ones I added this time:

  • Use exceptions to indicate a broken semantic contract.
  • Throw exceptions on abnormal conditions.
  • Use checked exceptions for conditions that client code may reasonably be expected to handle.
  • Throw a different exception type for each abnormal condition.

Tuesday, 9/12/2000 - Added pages for more guidelines, using primarily handout material from my Object Design Workshop. See the Table of Contents. Here are the ones I added:

  • Appreciate the significance of the interface.
  • Use interfaces to say "What objects can do for you" or "What you can do with objects".
  • Use interface extension to decouple services or restrict the semantic contract.
  • Consider creating convenience implementations of your interfaces.
  • Use abstract classes to provide default behavior or define a family.
  • Understand the role of intent in design patterns.
  • Appreciate the significance of the semantic contract.
  • Keep methods, objects, and packages focused.
  • Strive for interfaces that are minimal and complete.
  • Simplify, simplify, simplify.
  • Make the semantic contract as explicit as possible.
  • Make sure subtypes fulfill the semantic contracts of their supertypes.
  • Avoid optional clauses in semantic contracts.
  • Use deprecation, not a broken contract, if you have designer's regret.

Friday, 9/8/2000 - Added pages for more guidelines, using primarily handout material from my Object Design Workshop. I'm trying to get the material from the seminar up first, then I'll start fleshing out the text. See the Table of Contents.

  • Use Messengers to transmit information.
  • Use Immutables to represent values of abstract data types.
  • Use classes primarily to serve as blueprints for objects.
  • Use classes to say "What objects are."
  • Use class extension to model permanent IS-A relationships.
  • Prefer polymorphism over instanceof or reflection
  • Use composition enlist the help of other objects.
  • Understand the difference between inheritance and composition.

  • Use Flyweights for pluggable nuggets of behavior
