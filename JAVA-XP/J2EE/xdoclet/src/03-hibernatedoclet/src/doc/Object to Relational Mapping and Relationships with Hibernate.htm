Object to Relational Mapping and Relationships with Hibernate

 

There are several ways to persist data with Java.  The persistence layer is one of the most important layers in a Java application.  In most cases the persistence store is a relational database.  The persistence layer is often not given enough attention in my opinion.  This article will touch on some well-known persistence frameworks used in Java.   Then we will look at some new persistence frameworks such as Hibernate.  Lets start by looking at some of the more well-known frameworks and APIs available.

 

JDBC

 

Most Java developers start with code that includes JDBC to communicate to a back-end database.  This type of coding might be improved by using the DAO (Data Access Object) pattern that was written in-house and took significant time to develop.  Some implementations are written better than others.  If you really stop and think about it JDBC is really just a low-level connection to the database.  Writing code that includes the JDBC API is an important core Java skill.  However, writing SQL and tightly coupled DAO objects in a large application can be hard to maintain over time.  Most Java development teams understand how to program with JDBC, which attributes to the popularity of this API.  I believe that this persistence mechanism is the most widely used implementation today for better or worse. 

 

EJB

 

EJB is generally the second persistence API that most Java developers start to investigate.  To be more specific this involves using entity beans as a way to persist information to a database.  EJB with entity beans sounded like it might be the silver bullet of persistence framework with Java.  I mean there was great marketing behind this (sigh).  However, there was a catch.  First, you have to purchase an EJB container for a substantial price.  You can also obtain a better container that is free and open source such as JBoss - IMHO.  Another downside to entity beans is the time it takes to fully understand the EJB specification.  You will want to have an expert understanding of this API before coding with EJBs.  Also, you need to be aware that each container requires proprietary deployment descriptors outside of ejb-jar.xml. Furthermore, performance and support for the commercial containers are poor in most cases.  These are just some of the complexities that make Java developers continue to develop with JDBC.  Probably the most relevant components of EJB might be stateless-Session beans and messaging beans.

 

More Persistence Frameworks

 

After oscillating between these two specifications you might not feel either was a wonderful solution. Therefore, the development teams continue coding straight JDBC implementations and DAO objects. Recently, JDO started to get some traction as a useful API.  However, there were more hindrances once you scratched the surface with JDO.  Some companies attempting to provide JDO implementations deviated from the specification in some cases.  Furthermore, post byte-code compilers did not seem like a good option to Java developers. Therefore, most people are not using the JDO API for now.  This could change as the in the future as JDO matures.

 

 

What should you do when the APIs above seem like they might not satisfy your persistence needs?  There might be some more options available to you if you do not want to go mainstream.  Some of the emerging names include Hibernate and OJB.  These frameworks include some important differences from JDBC, EJB, and JDO. These persistence frameworks do not require a container, provide easy custom expressive and ODMG3-style APIs, they are open source, and they usually have good documentation with a strong underground following.  They are proprietary, but do you believe that the specifications discussed above are non-proprietary standards?  Please let me know what you think.

Hibernate

 

Lets start looking at one of these available persistence frameworks.  The framework that I have chosen to discuss is Hibernate.  While I think that OJB is also in the same category, I chose Hibernate due to the amount of documentation provided.  I know other people have used OJB with great success.  Anyways, Hibernate or OJB should provide everything you want from a persistence layer.  Visit http://hibernate.bluemars.net/4.html for a full list of Hibernate features.

 

I also want to quickly speak about the environment that I am working with.  I am using Hibernate 2.0 RC1 (2.0 beta 5) on Tomcat 4.1.18 in development and WebSphere Application Server 5.0.1 in staging and production.  This should work on other containers as well, but I have not personally tested them.  I am also using IBM DB2 with both application servers.  There are articles available on the Hibernate website for configuring this framework with JBoss.  Most application servers and databases should not have a problem interoperating with Hibernate.

Configuring Hibernate

 

Now that I got most of my comments out of the way above, I can jump into the code with you.  Click here for the source files in a zip format so that you can inspect them further.  There is also a DDL for the tables in the zip file.  The first file that to investigate is the hibernate.properties file.  Place this file in your classpath so that Hibernate can find it at runtime.  This file contains the runtime settings that Hibernate will use when it is started up.  Here is an example:

hibernate.show_sql=true

#pool ( 0 = fail, 1 = block, 2= grow)

hibernate.dbcp.ps.whenExhaustedAction 1

# The maximum number of milliseconds that the pool will

# to be returned before throwing an exception, or -1 to

# wait indefinitely.

hibernate.dbcp.ps.maxWait 120000

hibernate.dbcp.ps.maxIdle 100

 

  

The first set of properties specify the database driver used to get an underlying JDBC connection, the JDBC URL, user credentials, a default schema to use for the mapping documents, and a dialect.  This dialect will instruct Hibernate how to optimize the runtime settings for the database that you are using.  Refer to the Hibernate reference guide for other dialect settings for your database. Finally, the hibernate.show_sql property is enabled so that we can see what Hibernate is doing with SQL for debugging.  If you need to switch database implementations then just update this file. 

 

The rest of the properties specify settings for connection pooling.  This properties file specifies the Jakarta DBCP connection pool from the commons library.  Hibernate also can work with other connection pools such as C3PO.  Check the Hibernate resources for more information.

 

Creating Hibernate Persistence Objects

 

Now that the Hibernate environment is configured we now need some objects and mapping documents for Hibernate to work with.  Lets start with the classes that are required for persistence.  We will look at some of the dbdemo.User class since it is fairly central to the design.  The User class is basically just a JavaBean formatted file with public accessors and mutators.  Here is the complete file:

 

package dbdemo;

 

import java.util.Date;

import java.util.Set;

 

/**

 * @hibernate.class table="Users"

 *

 * @author MEagle

 *

 * Represents a User

 */

public class User {

 

      private String userID;

      private String userName;

      private String password;

      private String emailAddress;

      private Date lastLogon;

      private Set contacts;

      private Set books;

      private Address address;

 

      /**

       * @hibernate.property column="EmailAddress" type="string"

       * @return String

       */

      public String getEmailAddress() {

            return emailAddress;

      }

 

      /**

       * @hibernate.property column="LastLogon" type="date"

       * @return Date

       */

      public Date getLastLogon() {

            return lastLogon;

      }

 

      /**

       * @hibernate.property column="Password" type="string"

       * @return String

       */

      public String getPassword() {

            return password;

      }

 

      /**

       * @hibernate.id generator-class="assigned" type="string"

       *                      column="LogonID"

       * @return String

       */

      public String getUserID() {

            return userID;

      }

 

      /**

       * @hibernate.property column="Name" type="string"

       * @return String

       */

      public String getUserName() {

            return userName;

      }

 

      /**

       * @param string

       */

      public void setEmailAddress(String string) {

            emailAddress = string;

      }

 

      /**

       * @param string

       */

      public void setLastLogon(Date date) {

            lastLogon = date;

      }

 

      /**

       * @param string

       */

      public void setPassword(String string) {

            password = string;

      }

 

      /**

       * @param string

       */

      public void setUserID(String string) {

            userID = string;

      }

 

      /**

       * @param string

       */

      public void setUserName(String string) {

            userName = string;

      }

 

      /**

       * @hibernate.set role="contacts" table="Contacts" 

       *                        cascade="all" readonly="true"

       * @hibernate.collection-key column="User_ID"

       * @hibernate.collection-one-to-many class="dbdemo.Contact"

       * @return java.util.Set

       */

      public Set getContacts() {

            return contacts;

      }

 

      /**

       * @param set

       */

      public void setContacts(Set set) {

            contacts = set;

      }

 

      /**

       * @hibernate.set role="books" table="Book_User_Link"

       *                            cascade="all" readonly="true"

       * @hibernate.collection-key column="UserID"

       * @hibernate.collection-many-to-many

       *                            class="dbdemo.Book" column="BookID"

       * @return java.util.Set

       */

      public Set getBooks() {

            return books;

      }

 

      /**

       * @param set

       */

      public void setBooks(Set set) {

            books = set;

      }

 

      /**

       * @hibernate.one-to-one class="dbdemo.Address"

       * @return dbdemo.Address

       */

      public Address getAddress() {

            return address;

      }

 

      /**

       * @param address

       */

      public void setAddress(Address address) {

            this.address = address;

      }

 

}

 

Ant and XDoclet 

 

If you look closely at this file you will notice that there are some special Javadoc entries in the getters and the class documentation.  This is called Hibernatedoclet, which is a sibling to XDoclet. XDoclet is a tool that works with Apache Ant to create your deployment descriptors for you.  You should always use Ant for project deployment.  I think you will find that this is a real time saver if you are not already using it.  Unless you enjoy writing descriptor files in XML you should use XDoclet.  There are two reasons for this. First, this self-documents the code so that you can refer to descriptor file entries without opening the descriptor file.  Second, each time you deploy with Ant and the Hibernate XDoclet tools you can auto-generate your mapping documents.  Please refer to the build.xml file in the resources provided.  The target to run is called hibernate.  Also refer to http://xdoclet.sourceforge.net/tags/hibernate-tags.html for a complete list of Hibernate tags. All of the class files provided in this demo has XDoclet tags added to them, so I will leave this as an exercise for the user to self-discover. 

 

When the Ant script is run over the .java files it will produce the hbm.xml mapping documents. Make sure that you obtain the latest version of Xdoclet via CVS.  Xdoclet 1.2b3-dev was used when this article was written.  Here is a copy of the complete Ant script:

 

<!-- this file uses Apache Ant 1.5.3 beta 1 -->

<project name="Hibernate Example" default="about" basedir=".">   

      <!-- The location where your xdoclet jar files reside -->

      <property name="xdoclet.lib.home" value="c:/java_api/xdoclet-1.2b3/lib"/>

     

     

      <target name="clean" depends="init" description="removes all directories related to this build">

            <delete dir="${dist}"/>

      </target>

     

      <target name="init" description="Initializes properties that are used by other targets.">

            <property name="dist" value="dist"/>

      </target>

     

      <target name="prepare" depends="init,clean" description="creates dist directory">

            <echo message="Creating required directories..."/>

            <mkdir dir="${dist}"/>

      </target>  

     

      <target name="hibernate" depends="prepare"

        description="Generates Hibernate class descriptor files.">

        

       

            <taskdef name="hibernatedoclet"

                classname="xdoclet.modules.hibernate.HibernateDocletTask">

                <classpath>

                  <fileset dir="${xdoclet.lib.home}">

                      <include name="*.jar"/>

                  </fileset>

                </classpath>

            </taskdef>

 

            <!-- Execute the hibernatedoclet task -->

            <hibernatedoclet

                  destdir="."

                  excludedtags="@version,@author,@todo"

                  force="true"

                  verbose="true"

                  mergedir="${dist}">

 

                  <fileset dir=".">

                      <include name="**/dbdemo/*.java"/>

                  </fileset>

 

                  <hibernate version="2.0"/>

 

            </hibernatedoclet>               

      </target>

     

      <target name="about" description="about this build file" depends="init">

            <echo message="  Use this format for the arguments:"/>

            <echo message="      ant hibernate"/>

            <echo message=""/>                 

      </target>

</project>

 

 

If you are trying the examples in your environment and do not want to download XDoclet from CVS do the following: 

 

  1. Download and extract the examples.
  2. Extract hibernatedocletjars-1.2b3-dev.zip within the examples to a new directory.
  3. Edit the Ant script so that the property xdoclet.lib.home points to the directory that contains the jar files.

The XDoclet taskdef will also add the default values into the XML mapping document if you do not provide them.  Also, I could have had even less XDoclet tags if I wanted to omit the column names if they matched the properties name in the class.  However, I added them in for completeness.

 

Once the mapping documents have been created you will be ready to start using the Hibernate API to manipulate the classes.  Make sure that the mapping documents are in the classpath, which should probably be a one-to-one relationship to your class files.  The example files use the Hibernate custom expressive language API instead of the ODMG3 language style. 

 

Each of the examples includes a main method that calls a static run method.  The first Hibernate example, HibernateDemo.java, will add two new users and link them with a one-to-one relationship with an address. The second Hibernate example HibernateDemoOneToMany.java, will show how to use a one-to-many relationship with Hibernate,. Finally, the third example, HibernateDemoManyToMany.java, will show how to work with many-to-many relationships.  If you were actually trying these out I would recommend running them in the respective order.  You might also have to change database implementations if you are not using DB2.

 

I am choosing not to step through each of the examples because the code takes care of this in the comments.  Use the Hibernate Javadoc to reference classes that need more clarification.  These should be fairly straightforward to understand.  Manipulate the Java files to experiment with these relationships.

 

Tips and Pitfalls

 

Lets discuss some things that you should use with Hibernate.  First, make sure to review the documentation for Hibernate at http://hibernate.bluemars.net/5.html.  I constantly use the FAQ section and the reference documentation.  Also, check the forums for common problems. 

 

Use the Thread Local Session pattern when obtaining Hibernate Session objects.  You can read about this at http://hibernate.bluemars.net/42.html.  You can also find another example of this class in the provided example code.  It basically says that you should use a ThreadLocal class to store Session objects for each thread that is using it.  Each thread can see only the value associated with that thread, and is unaware that other threads may be using or modifying their own copies.  This is important because Hibernate's native API does not use the current thread to maintain associations between Session and Transaction or between Session and application thread.

 

One thing that always seems to trip developers up is that they do not properly handle resources. With Hibernate you will want to make sure that you properly flush and commit each Session object when persisting information.  Also make sure that you release, or close, the Session object when you are finished working with it.  You may also have to release the underlying connection if you choose to manually pass the connection into the Session.  Make sure that you issue session.close().close() to release the underlying connection back to the connection pool and then release the session.  If you do not pass a connection into the session you only need to close the session. 

 

I hope that this article has helped to get you on your way to exploring new persistence solutions such as Hibernate. Hibernate is a great alternative to JDBC and EJB.
